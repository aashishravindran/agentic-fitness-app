## Project Structure

All source code lives under `agentic-fitness-app/`. Commands should be run from that directory unless otherwise noted.

```
agentic-fitness-app/
├── main.py              # CLI entry point (ingest, query, run, db commands)
├── graph.py             # LangGraph workflow: build_graph(), run_workout(), run_onboard()
├── state.py             # FitnessState TypedDict (single source of truth for graph state)
├── llm.py               # LLM provider abstraction (Gemini/OpenAI/Ollama/Bedrock/DeepSeek)
├── feature_flags.py     # Toggle graph nodes on/off at import time
├── config.py            # Auto-loads .env on import
├── ingest.py            # ChromaDB ingestion for creator philosophy markdown files
├── db_utils.py          # Direct SQLite utilities (bypasses LangGraph checkpoint API)
├── agents/              # All LangGraph node implementations
│   ├── supervisor.py    # Entry node: routing, safety overrides, LLM-based decisions
│   ├── workers.py       # Four specialist workers (iron, yoga, hiit, kb)
│   ├── recommender.py   # Persona recommender (onboarding)
│   ├── finalize_workout.py  # Post-workout: apply fatigue, increment counters, update history
│   ├── decay.py         # Time-based fatigue decay (disabled by default)
│   ├── history_analyzer.py  # History-based fatigue analysis (disabled by default)
│   ├── greeting.py      # Dashboard greeting generator
│   ├── log_rest.py      # Rest day fatigue reduction
│   ├── recovery_worker.py   # Recovery workout generator
│   ├── retriever.py     # ChromaDB vector search (Ollama embeddings)
│   ├── trainer.py       # Shared LLM model accessor
│   └── workout_utils.py # inject_exercise_ids() helper
├── backend/
│   ├── main.py          # FastAPI app + WebSocket handler (/ws/workout/{user_id})
│   ├── routes/          # REST endpoints: onboard, workout, history, settings, status
│   └── services/workout_service.py  # Async wrapper around graph.py functions
├── shared/schemas.py    # Pydantic schemas for API responses
├── creators/            # Markdown files defining coach philosophies (RAG source)
├── creator_db/          # ChromaDB vector store (generated by ingest)
├── checkpoints/         # SQLite LangGraph checkpoint DB (per-user state persistence)
└── tests/               # pytest tests
```

## LLM Configuration

Set `LLM_PROVIDER` to force a backend, or leave unset for auto-detection (Gemini > OpenAI > DeepSeek > Ollama):

| Provider | Env var(s) |
|---|---|
| `gemini` | `GOOGLE_API_KEY` or `GEMINI_API_KEY` |
| `openai` | `OPENAI_API_KEY` |
| `deepseek` | `DEEPSEEK_API_KEY` |
| `ollama` | `OLLAMA_BASE_URL`, `OLLAMA_MODEL` |
| `bedrock` | AWS credentials, `BEDROCK_MODEL` |

Optional model overrides: `GEMINI_MODEL`, `OPENAI_MODEL`, `OLLAMA_MODEL`, `DEEPSEEK_MODEL`.

## Architecture: LangGraph Workflow

The core is a `StateGraph[FitnessState]` built in `graph.py`. The flow:

```
Entry → [recommender (if not onboarded)] → supervisor → [decay] → [history_analysis] → pre_route
                                                                                            ↓
                                               END ← finalize_workout ← iron/yoga/hiit/kb_worker
                                                                    ← recovery_worker
```

**Key design decisions:**

- **Supervisor is the safety governor**: It enforces two hard overrides before any LLM call — weekly frequency cap (`workouts_completed_this_week >= max_workouts_per_week` → `end`) and fatigue threshold (`max fatigue > 0.8` → `recovery_worker`). The supervisor short-circuits LLM calls when user input has no complaints or persona-switch keywords (`needs_llm_reasoning()`).

- **Graph interrupts after training workers**: The graph pauses after `iron_worker`/`yoga_worker`/`hiit_worker`/`kb_worker` (before `finalize_workout`), allowing the user to log sets via WebSocket `LOG_SET` messages. `finalize_workout` applies RPE-based fatigue, increments the weekly counter, and appends to `workout_history`.

- **Per-user persistence via SQLite**: Each user maps to a `thread_id` in the LangGraph `SqliteSaver` checkpoint DB. `db_utils.py` provides direct SQLite access (used to read state without triggering LangGraph's checkpoint loader, which can fail on schema mismatches).

- **Feature flags at import time**: `feature_flags.py` controls which nodes are added to the graph. `ENABLE_DECAY` and `ENABLE_HISTORY_ANALYZER` are off by default. Changing these requires restarting the server.

- **RAG for creator philosophies**: `creators/*.md` files are chunked and embedded into ChromaDB via `ingest.py` (requires Ollama with `mxbai-embed-large`). Workers call `retrieve_worker_philosophy(creator_name)` to ground workout generation. The retriever is Ollama-only (no alternative embedding provider).

- **Workers use pydantic-ai `Agent`**: Each worker creates a `pydantic_ai.Agent` with a structured `result_type` (e.g., `StrengthWorkoutPlan`). Agents run synchronously via `loop.run_until_complete()` inside LangGraph nodes (which are sync functions).

## WebSocket Protocol

Frontend connects to `ws://localhost:8000/ws/workout/{user_id}`. Message types:

- **Client → Server**: `USER_INPUT`, `LOG_SET`, `APPROVE_SUGGESTION`, `RESUME`, `FINISH_WORKOUT`, `RESET_USER`, `RESET_FATIGUE`, `RESET_WORKOUTS`, `LOG_REST`
- **Server → Client**: `AGENT_RESPONSE` (with `state`, `workout`, `is_working_out`), `ERROR`

After creating or modifying any feature, always validate by running the test suite:

```bash

```

When asked to "test this out", run `pytest tests/` to verify all tests pass. The e2e and integration tests use mocked workers (no LLM required). Only `test_graph.py` and `test_trainer.py` require a live LLM.

## Adding a New Persona

1. Create `creators/<creator_name>.md` with the training philosophy.
2. Re-run `python main.py ingest` to update ChromaDB.
3. Add a worker function in `agents/workers.py` with a matching Pydantic `WorkoutPlan` model.
4. Register the new worker node in `graph.py` (`build_graph()`).
5. Add the persona to the `Literal` types in `state.py` and `shared/schemas.py`.
6. Update the routing maps in `supervisor.py` (`persona_to_worker`) and `graph.py` (`route_after_supervisor`).
